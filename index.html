Why Hono is fast — architecture

Hono's performance comes from a combination of a purpose-built routing engine, zero dependencies, and being designed for Web Standards APIs from the ground up.

RegExpRouter vs Express routing
Hono — RegExpRouter
Hono compiles all registered routes into a single RegExp at startup. When a request arrives, one regex match against the URL determines the handler — O(1) regardless of route count. Inspired by the same technique used in high-performance routing libraries. Route count doesn't degrade performance.
Express — Linear iteration
Express iterates through registered routes one by one, testing each pattern in order. This is O(n) — the more routes your app has, the slower each request. A real app with 100+ routes and complex middleware chains shows measurable overhead per request that compounds at scale.
Result: Hono routes ~100K-250K req/s on Bun. Express routes ~25K-40K req/s on Node.js. The gap is partly runtime (Bun vs Node), partly router design.

Hono vs Express — when to use each
Hono vs Express — different tools for different contexts
Use Hono when
Deploying to edge/serverless (Cloudflare Workers, Deno Deploy, AWS Lambda edge). Using Bun or Deno as runtime. Performance is a hard requirement (high RPS APIs, latency-sensitive endpoints). Building from scratch and want TypeScript-first. Want zero dependencies in your production bundle. Using multiple runtimes in the same project.
Use Express when
You have an existing Express codebase — migration cost rarely justifies the gain. Your team has deep Express expertise and established patterns. You need a specific middleware that only exists in the Express ecosystem. You're building a simple REST API where performance is not a bottleneck. Your tech org has Express standardized across projects.
Bottom line: For new projects in 2025, Hono is the better default for APIs and edge functions. For existing Express apps, the performance gain rarely justifies a rewrite unless you have specific throughput issues.
Hono on edge runtimes

Hono's biggest performance advantage is on edge runtimes where Express cannot run at all. Express requires Node.js APIs that V8 Isolates (Cloudflare Workers) and other edge environments don't provide.

Hono uses the Fetch API (Request, Response, Headers) which is the Web Standard supported natively in all modern runtimes. This means: identical code runs on Cloudflare Workers (V8 Isolates with ~0ms cold start), Bun (JavaScriptCore), Deno, and Node.js 18+ without changes.

FAQ — Hono performance
Yes — significantly. On Node.js, Hono benchmarks at ~100K-150K req/s vs Express's ~25K-40K req/s for a simple JSON endpoint. On Bun, Hono reaches ~150K-250K req/s. The gap comes from Hono's RegExpRouter (O(1) routing vs Express's O(n) linear iteration), zero npm dependencies, and Web Standards APIs without adapter overhead. Express was designed in 2010 and carries significant legacy. The practical gap matters most at scale — for a small team at 1K req/s, you'll never notice.
Key factors: (1) RegExpRouter — compiles all routes into one regex, O(1) matching regardless of route count. (2) Zero dependencies — no npm packages, smaller bundle, faster startup. (3) Web Standards — native Request/Response from Fetch API, no Node.js adapter overhead. (4) Edge-first design — optimized for V8 Isolates where cold start is critical. (5) Modern TypeScript codebase — no legacy patterns slowing things down.
Throughput ranking (Node.js, higher = better): Fastify ≈ Hono (~100-150K req/s) → Express (~25-40K req/s). Fastify and Hono are competitive on Node.js. On Bun or edge runtimes, Hono wins because Fastify doesn't support those environments. On Cloudflare Workers, only Hono (and similar Web-Standard frameworks) run — Express and Fastify require a Node.js compatibility shim.
Only if: (1) You're starting a new project — use Hono by default. (2) You're deploying to edge/serverless where Express can't run without adapters. (3) You have a specific throughput bottleneck that profiling shows is framework overhead. Don't switch if: you have a large existing Express codebase, your team knows Express deeply, or you rely on Express-specific middleware. The API surface is similar but not identical — migration requires work. TypeScript experience is significantly better in Hono.
